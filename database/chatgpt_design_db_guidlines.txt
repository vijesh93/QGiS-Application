Part 2 — Database Model Design

Let’s now design a clean, extensible PostGIS schema.

Goal:

host many GIS layers

easy extension

good performance

flexible metadata

Schema Overview
PostGIS database
│
├── core      (app metadata)
├── data      (GIS layers)
└── auth      (users & permissions)

Core Metadata Tables
Layer Registry

Central layer listing.

CREATE TABLE core.layers (
    id SERIAL PRIMARY KEY,
    name TEXT UNIQUE,
    title TEXT,
    schema_name TEXT DEFAULT 'data',
    table_name TEXT,
    geom_column TEXT DEFAULT 'geom',
    srid INTEGER DEFAULT 4326,
    min_zoom INTEGER,
    max_zoom INTEGER,
    active BOOLEAN DEFAULT TRUE
);


Backend reads this to know available layers.

Layer Categories
CREATE TABLE core.layer_categories (
    id SERIAL PRIMARY KEY,
    name TEXT UNIQUE
);


Used for UI grouping.

Layer Styles
CREATE TABLE core.styles (
    id SERIAL PRIMARY KEY,
    layer_id INTEGER REFERENCES core.layers(id),
    style JSONB
);


Store symbology configs.

Data Schema — GIS Tables

Each layer has its own table.

Example:

CREATE TABLE data.roads_bw (
    id SERIAL PRIMARY KEY,
    geom GEOMETRY(LineString, 4326),
    name TEXT,
    road_type TEXT
);


Important:

CREATE INDEX roads_bw_geom_idx
ON data.roads_bw USING GIST (geom);


Without spatial index = slow maps.

Auth Schema (Basic)
CREATE TABLE auth.users (
    id SERIAL PRIMARY KEY,
    email TEXT UNIQUE,
    password_hash TEXT
);


Enough to start.

Import Workflow
Step 1 — Import OSM roads example

Use:

osm2pgsql


or GDAL:

ogr2ogr


Example:

ogr2ogr -f "PostgreSQL" \
PG:"dbname=gis user=postgres" \
roads.shp \
-nln data.roads_bw

Step 2 — Register Layer

Insert metadata:

INSERT INTO core.layers
(name, table_name, title)
VALUES
('roads_bw', 'roads_bw', 'Road network BW');


Now backend can discover it.

Step 3 — View in TileServer

Configure tileserver to expose layer.

Frontend loads:

/tiles/roads/{z}/{x}/{y}.png
